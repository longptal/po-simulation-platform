Decision Required

Sprint 3 — Stakeholder priority request

SH

Stakeholder Agent says:

"I need to re-prioritize Feature Z (Executive Dashboard) ahead of Feature A (User Search). The CEO has a board demo next week and this is a strategic commitment we can't miss. I know it disrupts the sprint — but can we make this work?"

Context

Feature Z is currently in Sprint 4. The stakeholder wants it moved into current sprint. This means one of the existing commitments must be deferred or cut.

Choose your response

A

Accept the priority change

Move Feature Z into current sprint, defer Feature C (Reporting)

Stakeholder trust increases significantly

Feature C reporting will be delayed to Sprint 5

Sets precedent: stakeholder may expect mid-sprint changes

B

Negotiate and push back

Ask for more context, understand why it's urgent. Explore partial scope or MVP

Demonstrates professional PO pushback skills

May find a compromise (partial scope in current sprint)

Stakeholder may become frustrated if not handled tactfully

C

Decline and keep sprint as-is

Keep current sprint commitments. Feature Z stays in Sprint 4 per plan

Protects sprint integrity and team focus

Stakeholder trust may decrease significantly

Must explain reasoning clearly or seem rigid

Coach whisper: Before committing, try asking the stakeholder what specifically the CEO needs. Sometimes a partial solution is enough. Option B lets you gather more info.
