# Architecture Design — PO Simulation Platform

> **Approved**: 2026-04-08 | **Status**: Approved by stakeholder

---

## 1. Design Principles

1. **Modularity first** — mỗi domain nghiệp vụ là một module độc lập, có thể upgrade/test/deploy riêng
2. **Loose coupling via events** — modules giao tiếp qua Event Bus, không gọi trực tiếp
3. **Real-time feedback** — PO thấy impact mỗi decision ngay lập tức
4. **Deterministic flow** — XState là single source of truth, mọi transition đều có audit trail
5. **Configuration-driven** — scenarios là YAML, không hard-code logic nghiệp vụ vào code

## 3. Event Bus Architecture

### Decision: Hybrid — BullMQ (jobs) + Redis Pub/Sub (notifications)

```
PO submits decision
        │
        ▼
┌─────────────────────┐
│  DecisionEngine     │
│  (orchestrator)     │
│                    │
│  1. Validate input  │
│  2. Score decision  │ ← ScoringEngine (real-time)
│  3. Update metrics  │
│  4. Dispatch jobs  │
└────────┬────────────┘
         │ BullMQ Queue
         ▼
  ┌──────────────┐     ┌──────────────┐
  │ BA Job       │────►│ BAAgent      │
  │ (Sonnet 4)   │     │ Output: spec │
  └──────┬───────┘     └──────────────┘
         │ BullMQ Queue (sequential)
         ▼
  ┌──────────────────┐  Redis Pub/Sub
  │ Stakeholder Job  │──────────────┐
  │ (Haiku 4.5)      │              │
  └────────┬─────────┘              ▼
           │ Output: feedback  ┌──────────────┐
           │                   │ SSE Broadcaster│
           └──────────────────►│ (orchestrator) │
           │                   └───────┬───────┘
           ▼                           │ SSE
  ┌──────────────┐              ┌───────▼───────┐
  │ Checkpoint   │              │  web (Next.js)│
  │ (PostgreSQL) │              │  Zustand store │
  └──────────────┘              └───────────────┘
```

### Redis Pub/Sub Channels
| Channel | Publisher | Subscribers | Payload |
|---------|-----------|-------------|---------|
| `session:{id}:events` | orchestrator | web (SSE) | `{type, nodeId, metrics, agentOutput}` |
| `session:{id}:metrics` | orchestrator | web (SSE) | `{dau, revenue, nps, ...}` |
| `agent:job:complete` | workers | orchestrator | `{jobId, output, error}` |

## 4. Agent Pipeline (Sequential — BA → Stakeholder)

### Flow

```
┌────────────────────────────────────────┐
│  SessionManager (XState)                               │
│                                                         │
│  State: awaitingDecision                               │
│         │ (Career Mode: halts at event-day checkpoint) │
│         │                                             │
│         │ [PO submits: "I want to add dark mode"]      │
│         ▼                                             │
│         │                                             │
│  State: processingDecision                            │
│         │                                             │
│         │ │ DecisionEngine scores decision             │
│         │ │     → computeSprintScore()                  │
│         │ │     → Redis Pub/Sub: metrics update          │
│         │ │                                             │
│         │ │ FreeformEvaluator (inline LLM call)        │
│         │ │     → classify intent → map to decision tree  │
│         │                                             │
│         │ │ AgentDispatcher.add('ba', job)             │
│         │                                             │
│                      │ BullMQ                          │
│                      ▼                                 │
│                                                         │
│                 BAAgent.run()                          │
│                 Output: { spec: {...} }                │
│                      │ BullMQ                          │
│                      ▼                                 │
│                                                         │
│              StakeholderAgent.run()                    │
│              Input:  { poDecision, spec }              │
│              Output: { feedback, sentiment }            │
│                      │ BullMQ                          │
│                      ▼                                 │
│                                                         │
│  State: waitingForAgents                             │
│         │ (orchestrator waits for job completion)      │
│         ▼                                             │
│                                                         │
│  State: awaitingDecision (next event-day checkpoint)  │
└───────────────────────────────────────────────
```

**Career Mode model:** State machine only halts at event-day checkpoints.
Between checkpoints, time passes freely (skip days). PO fast-forwards through
uneventful simulated days -- similar to Career Mode in a football game.

### Agent Contract Pattern
- **Input/Output**: typed + Zod validated
- **Retry**: max 2 attempts with format instructions on failure
- **Circuit breaker**: 5 consecutive failures → agent paused, notification sent to web
- **Timeout**: 30s per agent (configurable per agent type)

## 5. State Synchronization

### Three layers of state

```
┌──────────────────────────────────────────┐
│  Zustand (UI Layer)                      │  ← Ephemeral
│  sessionId, currentNode, messages[]      │
│  synced from SSE stream                  │
├──────────────────────────────────────────┤
│  XState (Orchestration Layer)            │  ← Single Source of Truth
│  session machine states, actor refs      │
│  snapshot → PostgreSQL on every transition│
├──────────────────────────────────────────┤
│  PostgreSQL (Persistence Layer)          │  ← Durable
│  sessions, decisions, agent_jobs,        │
│  scenario_completions                    │
└──────────────────────────────────────────┘
```

### Checkpoint Strategy
- XState snapshot lưu vào `sessions.state_snapshot` sau **mỗi transition**
- State reconstruct bằng cách load session + scenario + decisions từ DB
- On startup: kiểm tra active sessions, resume actors từ snapshot

## 6. Scenario System

### DB-first loading

```
YAML file (scenarios/*.yaml)
       │
       ▼ (seed script, startup)
Zod validation (scenario.schema.ts)
       │
       ▼ (pass)
PostgreSQL → scenarios table (jsonb)
       │
       │ (session load)
       ▼
ScenarioLoader: load from DB
  → ScenarioValidated (typed)
  → cache in memory (Map<id, ScenarioValidated>)
  → serve to XState actor
```

## 7. Scoring — Real-time Trigger

### Every PO decision triggers:

```
DecisionEngine.computeSprintScore()
    │
    ├── computeDimensionScores()  — aggregate modifiers per dimension
    ├── computeWeightedTotal()    — weighted sum
    └── Redis Pub/Sub → session:{id}:metrics
                             → session:{id}:events
```

### Session end triggers:
```
ScoringEngine.computeScore()
    │
    ├── SessionScore (total + per-dimension)
    ├── CausalChains (decision → impact)
    └── CausalGraph
```

## 9. Module Ownership & Dependency Rules

| Module | Owns | Dependencies | Can be upgraded independently? |
|--------|------|-------------|-------------------------------|
| `apps/web` | UI state, components | `packages/shared` | ✅ Yes |
| `apps/orchestrator` | Session flow, XState, events | `packages/shared`, `packages/db` | ✅ Yes |
| `apps/workers` | Agent execution | `packages/shared`, `packages/db` | ✅ Yes |
| `packages/shared` | Types, schemas, scoring | None | ✅ Yes |
| `packages/db` | Schema, migrations | `packages/shared` | ⚠️ Migrations couple to schema changes |

### Dependency Rule
```
apps/*  ──► packages/shared
apps/*  ──► packages/db
packages/db ──► packages/shared (types only)

apps/*  ──✗──► apps/*  (no cross-app direct calls)
```

All inter-app communication goes through:
- **BullMQ** for async jobs (BA → Stakeholder)
- **Redis Pub/Sub** for real-time events
- **REST/SSE** for web ↔ orchestrator

## 10. Phase 2 Readiness

Kiến trúc này được thiết kế để thêm agents mới mà **không sửa code hiện có**:

```
AgentRegistry (plugin pattern)
    │
    ├── BAAgent        ── registered
    ├── StakeholderAgent ── registered
    ├── DesignerAgent  ── Phase 2: register()
    ├── DevAgent       ── Phase 2: register()
    └── CustomerAgent  ── Phase 2: register()
```

Mỗi agent implement interface `AgentContract<Input, Output>` và gọi `AgentRegistry.register()`. Orchestrator không cần biết agent nào đang chạy — chỉ dispatch jobs qua BullMQ với `agentRole` field.
